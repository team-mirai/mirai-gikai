# 沼津市版みらい議会 要件定義書

## 1. 概要

本ドキュメントは、`mirai-gikai` を fork した現行プロジェクトを
「沼津市版」の政治 Web サービスへ作り替えるための要件定義をまとめる。

2026年3月17日時点で `upstream` を
`git@github.com:team-mirai/mirai-gikai.git` とし、`upstream/develop`
最新を取り込んだ状態を前提にする。

## 2. 背景

現行の `mirai-gikai` は、国会の法案・議案を対象とした構成になっている。
一方で沼津市版では、対象を「沼津市議会の議案・会議情報・関連資料」に
切り替え、市民が市政を追いやすくすることが目的となる。

ただし、現行コードベースには以下のような国政前提の概念が広く含まれる。

| 現行概念 | 現行の意味 | 沼津市版での扱い |
| --- | --- | --- |
| `bills` | 国会法案・議案 | 沼津市議会の議案・請願・陳情・報告案件などの総称として再定義する |
| `diet_sessions` | 国会会期 | 沼津市議会の定例会・臨時会・委員会単位へ置き換える |
| `originating_house` | 衆議院/参議院 | 自治体議会には不適合のため再設計する |
| `shugiin_url` | 衆議院側の原典URL | 沼津市議会/沼津市の原典URLへ置き換える |
| `mirai_stances` | チームみらいの賛否・見解 | 沼津市版の運営主体または会派の見解として再定義する |
| トップの「本日の国会」 | 現在の国会会期表示 | 現在開催中の定例会/臨時会や直近の会議情報へ置き換える |

## 3. 目的

- 沼津市議会の議案や会議情報を、市民が理解しやすい形で公開する
- 行政・議会の原典資料にたどり着きやすくし、情報の透明性を高める
- 市民が「何が議論されているのか」「誰に影響があるのか」を短時間で把握できるようにする
- 既存 upstream の改善を取り込み続けられるよう、大規模な全面作り直しではなく段階的なローカライズで進める

## 4. 対象ユーザー

- 沼津市政に関心のある市民
- 沼津市内の当事者、事業者、保護者、学生など
- 地方政治を継続的に追いたいメディア・研究者・支援者
- コンテンツを登録・更新する運営メンバー

## 5. 前提方針

- fork 元の upstream 更新を今後も定期的に取り込む
- 初期段階では内部テーブル名やディレクトリ名を無理に全面改名せず、画面表示・用語・業務フローの整合を優先する
- 国政依存の項目は「そのまま流用する項目」と「後方互換を保ちつつ再定義する項目」を切り分ける
- 沼津市の公式情報を原典とし、要約や解説は原典URLと併記する
- まずは「議案を追えること」を最優先にし、AI インタビューや意見収集の高度機能は段階導入とする

## 6. スコープ

### 6.1 初期リリースの対象

- 公開 Web サイトの沼津市版への文言・構成変更
- 管理画面の議案/会期管理を沼津市議会向け運用に合わせて調整
- 議案一覧、議案詳細、公開管理、タグ管理、サムネイル管理
- 定例会/臨時会の管理と、トップページへの表示
- 原典URL、関連資料URL、議決結果など地方議会向けメタデータの管理
- 既存の難易度別コンテンツ表示が有効であれば継続利用する

### 6.2 後続フェーズの対象

- AI チャットの自治体議会向け最適化
- インタビュー機能、意見収集、公開レポート機能
- 議員別・会派別の賛否トラッキング
- 議事録、録画、委員会資料との自動連携
- スクレイピングや自動インポートの仕組み

### 6.3 初期リリースでは対象外

- 全国自治体への横展開を前提としたマルチテナント化
- 国会向け機能との完全な両立
- 自動収集基盤のフル実装
- 議員個人認証や高度な市民認証

## 7. 機能要件

### 7.1 公開 Web

#### 7.1.1 トップページ

- サービス名、説明文、ヒーローセクションを沼津市版へ変更する
- 現在開催中または直近の定例会/臨時会を表示する
- 注目議案、カテゴリ別議案、近日公開議案、過去会期アーカイブを表示する
- 「国会」「法案」など国政前提の文言を「市議会」「議案」などへ置き換える

#### 7.1.2 議案一覧

- 公開中の議案を一覧表示できる
- 会期、カテゴリ、公開状態、進行状況などで整理できる
- 将来的な検索・フィルタ追加を阻害しない構造にする

#### 7.1.3 議案詳細

- 議案名、分かりやすいタイトル、要約、本文、原典URLを表示できる
- 関連会期、提出者区分、委員会付託先、議決結果、公開日を表示できる
- 必要に応じて運営主体の見解を表示できる
- 難易度切り替え機能がある場合は市民向けの平易な解説として活用する

#### 7.1.4 アーカイブ

- 過去の定例会/臨時会単位で議案一覧を閲覧できる
- 「前回の国会」相当のUIは「前回の定例会/過去の会期」へ置き換える

### 7.2 管理画面

#### 7.2.1 会期管理

- 沼津市議会の定例会・臨時会を登録、編集、公開管理できる
- 現在アクティブな会期を1件指定できる
- 開催期間、表示順、説明文を管理できる

#### 7.2.2 議案管理

- 議案の作成、編集、複製、公開/下書き管理ができる
- タイトル、要約、本文、サムネイル、タグ、原典URLを編集できる
- 会期との紐付け、進行状況、議決結果、関連資料URLを管理できる

#### 7.2.3 コンテンツ管理

- 初学者向け/標準/詳細など複数難易度の解説を管理できる
- 将来的な AI 補助入力を見据えつつ、まずは手動編集を前提にする

### 7.3 運用

- 原典資料のURLと最終確認日時を保持できる
- 公開前プレビューが利用できる
- 管理者権限を持つユーザーのみが編集できる

## 8. データ要件

### 8.1 維持してよい既存概念

- `bills`: 内部実装上は当面維持し、UI 上は「議案」として扱う
- `bill_contents`: 議案ごとの説明コンテンツとして維持する
- `publish_status`: `draft` / `coming_soon` / `published` を継続利用する
- `tags`: 議案カテゴリ分類に継続利用する

### 8.2 再設計が必要な概念

#### 8.2.1 会期

- `diet_sessions` は概念上「議会会期」に再定義する
- 名称候補は内部的には据え置きでもよいが、表示文言は国会前提を排除する
- 将来的に本会議と委員会を分けて管理する可能性を考慮する

#### 8.2.2 議案進行ステータス

現行の `originating_house` と国会向けステータスは自治体議会と整合しない。
初期要件として、以下のような地方議会向けステータス体系へ見直す。

- `scheduled`: 上程予定
- `submitted`: 提出済み
- `under_review`: 審査中
- `committee_review`: 委員会審査中
- `plenary_review`: 本会議審議中
- `approved`: 可決
- `rejected`: 否決
- `continued`: 継続審査
- `withdrawn`: 取り下げ

内部 enum 名や段階的移行方法は設計フェーズで確定する。

#### 8.2.3 原典リンク

- `shugiin_url` は用途不一致のため、自治体議会向けの汎用的な
  `source_url` または複数資料URL管理への移行が必要
- 議案ページ、議事日程、議決結果、会議録、録画URLを扱える設計が望ましい

#### 8.2.4 見解データ

- `mirai_stances` は「運営主体の見解」または「会派の見解」として再定義する
- 将来的に複数会派を扱う可能性がある場合、単一主体固定の設計は見直し対象とする

## 9. 非機能要件

### 9.1 保守性

- upstream の更新を取り込みやすい差分量に抑える
- 国政固有文言の置換を設定化または局所化し、今後の改修コストを下げる

### 9.2 可用性・運用性

- 管理者が手動で情報更新できる
- 原典URL切れや更新漏れを運用で検知しやすい構成にする

### 9.3 UX

- スマートフォンで読みやすい
- 難しい行政用語をそのまま並べず、要約と原典を併記する
- 「何の議案か」「誰に関係するか」「今どの段階か」を一画面で把握しやすくする

### 9.4 SEO・共有

- 議案詳細ページは OGP と共有サムネイルに対応する
- 議案ごとに安定したURLを持つ

## 10. 移行要件

- 既存 seed データの国会法案サンプルは、沼津市版のダミーデータまたは実データに置き換える
- UI 文言、メタデータ、説明文、導線から国政前提の表現を除去する
- 「本日の国会」「衆議院/参議院」などの表示箇所を洗い出して差し替える
- 既存の AI プロンプトや文面に含まれる「法案」「チームみらい」の前提は段階的に見直す

## 11. リスク

- 国政前提の enum や DB カラムが多く、単純な文言置換では不整合が残る
- 沼津市議会の公開データ形式次第では、手動運用の負荷が高くなる
- どの主体の見解を掲載するかが未確定だと、`mirai_stances` の扱いが曖昧になる
- upstream の進化速度が速いため、長期間の大規模改修ブランチは追従コストが高い

## 12. 要確認事項

- サービスの正式名称をどうするか
- 対象を「本会議の議案」までに絞るか、「委員会」「請願」「陳情」「報告案件」まで含めるか
- 原典データは手入力中心で始めるか、CSV/スクレイピング連携を初期から入れるか
- 見解を掲載する主体は誰か
- AI チャット、AI インタビュー、意見収集機能を初期リリースに含めるか
- 沼津市議会以外の周辺情報（議員、会派、予算、条例など）をどこまで対象にするか

## 13. 次フェーズで詰める設計論点

- 自治体議会向けのステータス enum と進行表示 UI の設計
- `diet_sessions` / `originating_house` / `shugiin_url` の移行方針
- 沼津市版 seed データと管理画面入力項目の具体設計
- 初期リリースで残す upstream 機能と、後続フェーズへ送る機能の切り分け
