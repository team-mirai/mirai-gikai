# 沼津市議会データ取り込み MVP 実装計画

## 1. 目的

`mirai-gikai` を沼津市版へ寄せる第一段階として、
沼津市議会の公式公開情報を継続的に取り込み、
公開サイトと管理画面に反映できる MVP を実装する。

本計画は、既存 upstream との差分を最小限に抑えつつ、
まずは「会期」「議案」「原典リンク」「市民向け要約」を
安定して運用できる状態を作ることを目標にする。

## 2. MVP の到達点

MVP 完了時にできるようにすること:

- 沼津市議会の定例会/臨時会を管理できる
- `本会議のお知らせ` から提出議案一覧を取り込める
- 議案 PDF を元に議案メタデータと原典リンクを登録できる
- 管理画面で要約・タイトル・タグ・公開状態を編集できる
- 公開サイトで議案一覧・議案詳細・会期別アーカイブを表示できる
- 議案詳細から原典 PDF、会議録検索、関連動画へ遷移できる

MVP ではまだやらないこと:

- 会議録全文の自動取り込み
- 発言単位の構造化
- 議員別賛否の自動抽出
- 完全自動スクレイピングのみでの無人運用

## 3. 実装方針

### 3.1 基本方針

- 既存 `bills` / `bill_contents` / `diet_sessions` を活かす
- 破壊的な rename より additive migration を優先する
- 原典の正本は外部 URL とし、サービス内には要約と構造化メタデータを保持する
- 取り込みはまず CLI ベースの半自動運用で始める
- 管理画面は「取り込み後の整形・公開管理」に集中させる

### 3.2 段階的移行方針

#### Phase A: 既存スキーマの延命

- `diet_sessions` は内部名を維持したまま「会期」として使う
- `originating_house` は暫定値固定または UI 非表示でしのぐ
- `shugiin_url` は新フィールド導入までの間、使用停止または代替扱いとする

#### Phase B: 地方議会向け情報の追加

- 原典 URL 群を格納する別テーブルを追加
- 議案番号や議案種別を追加
- 会議日単位の情報を持てるようにする

#### Phase C: 国会依存フィールドの整理

- `originating_house`
- 国会向け `bill_status_enum`
- `shugiin_url`

これらを地方議会向け設計へ移行する。
ただし MVP の初期着手ではここまで一気にやらない。

## 4. DB 変更案

### 4.1 既存テーブルへの追加

#### `bills`

追加推奨カラム:

- `bill_number`: `議第4号` などの議案番号
- `bill_kind`: `議案` `報告` `認定` `請願` など
- `proposer`: `市長` `議員提出` など
- `submitted_at`: 提出日
- `source_updated_at`: 元ページの更新確認日時
- `session_order`: 会期内での並び順
- `result_text`: `可決` `否決` `継続審査` など
- `committee_name`: 主たる付託委員会

理由:

- 沼津市議会では議案番号が実務上重要
- `name` だけでは議案一覧と PDF の対応付けがしづらい
- 会期別表示と検索にも必要

#### `diet_sessions`

追加推奨カラム:

- `slug`: URL 用識別子
- `session_type`: `regular` `extraordinary`
- `session_number`: 第12回など
- `fiscal_year`: 令和7年度相当の整理軸
- `announcement_url`: 本会議のお知らせ URL
- `schedule_url`: 会期予定 / 会期日程 URL

理由:

- 既存 `diet_sessions` は国会会期前提のため、地方議会向けに意味を持たせ直す必要がある
- 会期ごとの公開ページ URL を持っておくと再取得しやすい

### 4.2 新規テーブル

#### `bill_source_documents`

議案ごとに複数の原典を持つためのテーブル。

- `document_type`
- `title`
- `url`
- `source_format`
- `published_at`
- `fetched_at`
- `text_content`
- `metadata`

`document_type` の初期値候補:

- `bill_pdf`
- `schedule_pdf`
- `general_questions_pdf`
- `bill_questions_pdf`
- `minutes_search`
- `plenary_video`
- `committee_video`

#### `council_meetings`

会期配下の会議日程や委員会日程を管理する。

想定カラム:

- `id`
- `diet_session_id`
- `meeting_type`
- `name`
- `scheduled_at`
- `room`
- `committee_name`

- `video_url`

### MVP 時点では作らないが将来候補

- `council_members`
- `factions`
- `committees`
- `bill_meeting_relations`

## 5. 既存フィールドの扱い

### 5.1 `originating_house`

MVP では以下のどちらかで対応する。

1. 固定値を保存し、公開 UI では完全に非表示
2. フォーム上も非表示にして server 側で暫定値を補完

推奨は 2。
地方議会では意味のない入力を管理画面に出さない方がよい。

### 5.2 `shugiin_url`

MVP では新規入力をやめ、`bill_source_documents` へ移行する。

既存実装が `shugiin_url` を参照している箇所は、
段階的に「原典リンク一覧を引く実装」に置き換える。

### 5.3 `bill_status_enum`

MVP では全面移行せず、以下のマッピングで暫定運用する。

- 公開前: `preparing`
- 提出済み: `introduced`
- 委員会付託 / 審査中: `in_originating_house`
- 本会議審議中: `in_receiving_house`
- 可決: `enacted`
- 否決: `rejected`

この暫定運用は UI で地方議会向け文言へ変換して吸収する。

## 6. 取り込み実装の配置案

### 6.1 推奨配置

新規 CLI スクリプト群を `packages/seed` 配下に追加する。

理由:

- すでに Supabase 管理クライアント利用基盤がある
- CSV import / seed 実装が存在し、運用感が近い
- MVP では cron より手動実行 CLI の方が扱いやすい

想定構成:

```text
packages/seed/
├── numazu/
│   ├── fetch/
│   ├── parse/
│   ├── transform/
│   └── run-numazu-import.ts
```

追加スクリプト案:

```json
{
  "scripts": {
    "numazu:import": "tsx numazu/run-numazu-import.ts",
    "numazu:dry-run": "tsx numazu/run-numazu-import.ts --dry-run"
  }
}
```

### 6.2 なぜ `admin` 側の Server Action にしないか

- HTML / PDF 取得と抽出はバッチ処理向き
- 失敗時のリトライやログ確認を CLI の方が扱いやすい
- MVP では管理画面からワンクリック実行より、まず安定処理が重要

## 7. 取り込みフロー

### 7.1 会期取り込み

ソース:

- `annai/yotei.htm`
- `annai/oshirase.htm`

処理:

1. 年間予定ページから今後の定例会一覧を抽出
2. `本会議のお知らせ` の現在会期と照合
3. `diet_sessions` を upsert

### 7.2 議案一覧取り込み

ソース:

- `annai/oshirase.htm`

処理:

1. 提出議案リンク群を抽出
2. `議第xx号` などの議案番号を抽出
3. リンク URL を `bill_source_documents` に保存
4. `bills` を upsert

upsert の照合キー候補:

- `diet_session_id + bill_number`

### 7.3 PDF 取り込み

ソース:

- 各議案 PDF

処理:

1. PDF を取得
2. `pdftotext` 相当でテキスト抽出
3. 抽出テキストを `bill_source_documents.text_content` に保存
4. 下書きの `bill_contents.summary/title/content` を生成

### 7.4 会議日程取り込み

ソース:

- 会期日程 PDF
- `annai/nittei.htm` の年度 PDF

処理:

1. 本会議日程を `council_meetings` に登録
2. 委員会日程が取れれば委員会日程も登録
3. 議案詳細と会議ページから関連会議を表示

### 7.5 会議録・動画リンク取り込み

ソース:

- AmiVoice 会議記録検索
- DiscussVision / YouTube

処理:

1. まずは会期単位または固定導線 URL を登録
2. 議案詳細に「会議録で探す」「録画を見る」を表示
3. 議案個別紐付けは後続フェーズで改善

## 11. テスト方針

- HTML parser
- PDF 抽出テキスト parser
- 文字列正規化
- 議案番号抽出
- 日付変換
- Supabase への upsert
- `bill_source_documents` 連携
- `diet_sessions` と `bills` の紐付け

公式 HTML が変わると parser が壊れうるため、
代表的な HTML / PDF 抽出テキストを fixture 化しておく。

## 12. リスクと対策

### 12.1 公式 HTML 変更

対策:

- parser は見出しテキスト基準を優先
- fixture テストを作る

### 12.2 PDF レイアウト崩れ

対策:

- まずは議案番号と題名だけ取れればよい設計にする
- 本文要約は人手修正を前提にする

### 12.3 著作権

対策:

- 原典 URL リンク方式
- 抽出テキストの一般公開は避け、内部処理用に留める

### 12.4 upstream 追従コスト

対策:

- additive migration を優先
- UI 差し替えを局所化

## 13. 次の実装タスク

最初の1本としては、次を着手対象にするのがよい。

1. `bill_source_documents` と `council_meetings` の DB 設計を確定する
2. `packages/seed/numazu/run-numazu-import.ts` の骨組みを作る
3. `oshirase.htm` parser を実装する
4. 管理画面の議案編集から `originating_house` と `shugiin_url` を外す
5. 議案詳細に原典リンク一覧を出す

この順なら、最小の実装で沼津市議会の実データをサイトへ載せ始められる。
